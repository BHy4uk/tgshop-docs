# Інструкція для власника магазину

Цей гайд пояснює, як налаштувати Telegram-магазин з нуля, як ним користуватися щодня і що саме робити в кожному основному модулі.

Магазин працює у двох інтерфейсах:

- **Telegram-бот** — для входу, швидких дій, оперативних перевірок і частини адмін-сценаріїв
- **Telegram WebApp** — для повноцінного керування каталогом, оплатами, доставкою, клієнтами, кампаніями, аналітикою, підпискою і виглядом магазину

!!! tip "Коротко"
    Бот потрібен для швидкого доступу і термінових дій. WebApp — це головний робочий кабінет власника магазину, де ви проводите основну частину часу.

---

## 🚀 Швидкий старт

Якщо ви хочете максимально швидко підготувати магазин до прийому замовлень, виконайте саме цей порядок:

1. Відкрийте свого Telegram-бота і введіть `/admin`.
2. Якщо система просить прийняти юридичні документи — прийміть їх.
3. Переконайтеся, що вам доступні розділи `📦 Товари`, `📋 Замовлення` і `⚙️ Налаштування`.
4. Якщо доступу немає — виконайте `/myid` і передайте свій Telegram ID власнику магазину або платформному адміну.
5. Створіть категорії товарів.
6. Додайте товари з фото, цінами і описами.
7. Увімкніть хоча б один спосіб оплати і зробіть тестову оплату.
8. Якщо плануєте відправки Новою поштою — увімкніть модуль і внесіть дані відправника.
9. Перевірте тексти, кнопки і базові налаштування магазину.
10. Зробіть тестове замовлення як покупець і проведіть його до кінця.

!!! warning "Порада для першого запуску"
    Не починайте з промокодів, кампаній чи аналітики. Спочатку доведіть до стабільного стану базовий ланцюжок:

    `товар → кошик → checkout → оплата → замовлення → відправка`

    Тільки коли цей ланцюжок працює без збоїв, має сенс підключати модулі росту.

## 1. Хто за що відповідає

### Власник магазину або адміністратор магазину

Власник магазину керує всім, що стосується щоденної роботи:

| Зона відповідальності | Що саме |
|---|---|
| Каталог | Категорії, товари, варіанти, фото, залишки |
| Продажі | Замовлення, підтвердження оплат, статуси |
| Оплати | Підключення і налаштування способів оплати |
| Доставка | Модуль Нової пошти, створення ТТН |
| Контент | Тексти, назви кнопок, зовнішній вигляд каталогу |
| Клієнти | Промокоди, кампанії, аналітика, cross-sell |
| Управління | Команда адміністраторів, підписка, режим доступу |

### Платформний адміністратор

Платформний адміністратор відповідає за те, що **не керується** звичайним інтерфейсом магазину:

- Технічне розгортання і підтримку інфраструктури
- Підключення бота та середовища
- Критичні доступи і відновлення доступу
- Загальні технічні питання, які не вирішуються через ваші налаштування

!!! info "Коли звертатися до платформного адміна"
    Звертайтеся, якщо бот не відповідає взагалі, магазин не відкривається, є проблема з доступом на рівні платформи або потрібні зміни поза межами адмінки.

## 2. Основні команди та перший вхід

### Команди, які треба знати

| Команда | Що робить |
|---|---|
| `/admin` | Відкриває адмін-меню магазину |
| `/guide` | Відкриває коротку інтерактивну інструкцію прямо в боті |
| `/myid` | Показує ваш Telegram ID для надання доступу |

### Що відбувається під час першого входу

Коли ви вперше заходите через `/admin`, система може послідовно попросити:

1. Прийняти **політику конфіденційності**
2. Прийняти **правила використання платформи**

Після прийняття обох документів відкриється адмін-меню. Це відбувається один раз — при наступних входах ви одразу потрапляєте в меню.

### Якщо бачите повідомлення «Немає доступу»

Це означає, що ваш Telegram ID **ще не додано** до списку адміністраторів цього магазину.

**Що робити:**

1. Виконайте `/myid` у чаті з ботом
2. Скопіюйте числовий Telegram ID, який бот покаже
3. Передайте цей ID власнику магазину або платформному адміну
4. Попросіть додати вас у список адміністраторів

!!! tip "Важливо"
    Telegram ID — це числовий ідентифікатор (наприклад, `123456789`), а не ваш юзернейм (`@username`). Не плутайте ці два значення.

### Де краще працювати: бот чи WebApp

**Бот підходить для:**

- Швидкого входу в адмінку
- Перегляду і обробки замовлень
- Оперативних дій на ходу
- Короткої довідки через `/guide`

**WebApp підходить для:**

- Регулярної роботи з каталогом і товарами
- Імпорту каталогу з файлів
- Налаштування промокодів, кампаній, аналітики
- Роботи з клієнтами та сегментами
- Підписки, оплат, рахунків і договорів
- Оформлення зовнішнього вигляду магазину
- Масових і швидких змін

!!! tip "Практичне правило"
    Якщо треба зробити одну швидку дію — починайте з бота. Якщо треба керувати магазином системно — переходьте у WebApp.

## 3. Як влаштоване керування магазином

### Telegram-адмінка (бот)

Після команди `/admin` у боті ви побачите меню з кнопками:

- `📦 Товари` — перегляд, додавання і редагування товарів
- `📂 Категорії` — управління категоріями каталогу
- `📋 Замовлення` — оперативна робота з замовленнями і фільтри за статусами
- `📊 Статистика` — якщо модуль аналітики увімкнено на вашому тарифі
- `📣 Кампанії` — якщо модуль кампаній увімкнено
- `⚙️ Налаштування` — загальні налаштування магазину

### WebApp — хаб керування

У WebApp доступні вкладки, але їхній склад залежить від тарифу і активних модулів. Повний набір:

| Вкладка | Призначення |
|---|---|
| **Товари** | Список каталогу, пошук, швидке редагування |
| **Імпорт** | Завантаження каталогу з CSV/XLSX |
| **Промокоди** | Створення і керування знижками |
| **Кампанії** | Розсилки по сегментах клієнтів |
| **Аналітика** | Виторг, конверсія, популярні товари |
| **Крос-продажі** | Пари товарів, які часто купують разом |
| **Клієнти** | База клієнтів, сегменти, LTV |
| **Налаштування** | Оплати, доставка, вигляд, тексти, команда |

### Секції всередині «Налаштування»

Налаштування у WebApp поділені на логічні блоки:

- **Підписка й оплати** — тариф, рахунки, способи оплати клієнтів
- **Вигляд** — тема, кольори, шрифт, логотип, банер
- **Автоматизація** — нагадування про кошик, модулі магазину
- **Операційка** — кабінет покупця, Нова пошта
- **Команда** — адміністратори магазину
- **Тексти й кнопки** — стартові тексти, назви кнопок, повідомлення

!!! info "Якщо якоїсь вкладки або блоку немає"
    Це зазвичай означає одне з трьох:

    1. Модуль ще не увімкнено в секції «Автоматизація → Модулі»
    2. Поточний тариф не дозволяє цю можливість
    3. Функція доступна тільки в певній збірці магазину

## 4. Повне налаштування магазину з нуля

Нижче наведений безпечний порядок, у якому варто налаштовувати магазин. Якщо дотримуватися цієї послідовності, ви уникнете більшості типових проблем на старті.

### Крок 1. Перевірте доступ і ролі

Перед будь-якими налаштуваннями переконайтеся:

- [ ] Ви заходите саме в потрібного бота
- [ ] Команда `/admin` відкриває адмін-меню
- [ ] Ви бачите мінімум «Товари», «Замовлення» і «Налаштування»

Якщо доступу немає — спочатку вирішіть це через `/myid` і додавання в адміни (див. [розділ 2](#2)).

### Крок 2. Визначте, як будете продавати

Ще до внесення товарів визначте:

- Чи будуть товари **в наявності**, чи частина буде **під замовлення**
- Чи потрібні **варіанти** (розміри, кольори)
- Які **способи оплати** ви реально будете використовувати
- Чи потрібна **Нова пошта** відразу

!!! warning "Правило для товарів під замовлення"
    Товари `під замовлення` не можна залишати тільки з післяплатою. Для них **обов'язково** має бути хоча б один онлайн-спосіб оплати (Monobank Merchant, оплата за посиланням або за реквізитами). Якщо залишити тільки післяплату, checkout для таких товарів буде заблокований.

### Крок 3. Створіть структуру каталогу

Спочатку створіть категорії, а вже потім додавайте товари:

- Головні категорії за групами товарів
- Товари всередині категорій
- Варіанти або розміри — тільки там, де вони реально потрібні

### Крок 4. Додайте товари і перевірте картки

На цьому етапі перевірте, що в кожного товару заповнено:

- [ ] Зрозуміла назва
- [ ] Короткий опис без зайвого шуму
- [ ] Коректна ціна
- [ ] Правильна категорія
- [ ] Хоча б одне фото (бажано 2-3)
- [ ] Правильний режим складу (`tracked` або `untracked`)
- [ ] Активність товару — він має бути **увімкнений** у каталозі

### Крок 5. Налаштуйте оплату

Увімкніть **хоча б один** реальний спосіб оплати і зробіть тестове замовлення. Детальна інструкція для кожного способу оплати — нижче в [розділі 6](#6_1).

!!! warning "Не вважайте оплату налаштованою, поки самі не перевірили"
    - Клієнт бачить потрібний варіант оплати в checkout
    - Інструкція по оплаті зрозуміла без додаткових пояснень
    - Замовлення переходить у правильний статус
    - Ви можете обробити це замовлення в адмінці

### Крок 6. За потреби налаштуйте Нову пошту

Якщо плануєте реальні відправки через Нову пошту — краще зробити це **до** першого живого продажу. Повна інструкція — в [розділі 7](#7_1).

### Крок 7. Перевірте тексти, кнопки і вигляд

До запуску переконайтеся:

- У стартових і службових текстах немає тимчасових або тестових фраз
- Кнопки названі так, як ви хочете бачити їх у магазині
- Зовнішній вигляд каталогу відповідає вашому бренду

### Крок 8. Зробіть повний тест як покупець

Перевірка повинна пройти **весь шлях**, а не лише відкриття каталогу:

1. Відкрити магазин як клієнт (через інший Telegram-акаунт або попросити друга)
2. Переглянути каталог і категорії
3. Додати товар у кошик
4. Пройти checkout
5. Обрати оплату і провести її
6. Довести замовлення до потрібного статусу в адмінці
7. Якщо є доставка — перевірити дані для відправки

!!! tip "Тільки після повного тесту"
    Магазин реально готовий до старту тільки після того, як ви самі пройшли весь шлях від перегляду каталогу до завершеного замовлення. Не пропускайте цей крок.

## 5. Каталог: товари, категорії, фото, варіанти, залишки

### 5.1. Де керувати каталогом

**У боті:** `/admin` → `📦 Товари` або `📂 Категорії`

**У WebApp:** вкладка `Товари` або `Імпорт` (якщо модуль доступний)

Для щоденної роботи з каталогом зручніше WebApp — там є пошук, список товарів і швидке редагування.

### 5.2. Категорії

Категорії впливають на навігацію у каталозі і на зручність роботи з великим асортиментом.

**Рекомендований порядок:**

1. Створіть основні категорії
2. Перевірте, що назви зрозумілі покупцю (не внутрішні коди, а людські назви)
3. Лише після цього прив'язуйте товари

!!! tip "Скільки категорій потрібно"
    Якщо асортимент невеликий (до 20 товарів) — достатньо 2-4 категорії. Якщо великий — не змішуйте різні логічні групи товарів в одній категорії. Покупець має знайти потрібне за 1-2 кліки.

### 5.3. Створення товару

Під час створення товару заповніть ці поля:

| Поле | Пояснення |
|---|---|
| **Назва** | Як покупець бачитиме товар у каталозі |
| **Опис** | Короткий, без зайвого. Покупець читає 2-3 речення |
| **Ціна** | У гривнях. Перевірте, що немає зайвих нулів |
| **Категорія** | До якої групи належить товар |
| **Фото** | До 5 фото. Перше — головне для картки |
| **Режим складу** | `tracked` (з обліком) або `untracked` (під замовлення) |
| **Варіанти** | Розміри, кольори — якщо є |
| **Активність** | Має бути увімкнена, щоб товар з'явився у каталозі |

### 5.4. Фото товарів

Один товар підтримує до **5 фото**.

!!! tip "Практичні правила для фото"
    - **Перше фото** — головне, найчистіше і найінформативніше. Саме воно показується в картці каталогу
    - Інші фото — деталі, кольори, ракурси, текстури
    - Рекомендується квадратний або вертикальний формат
    - Якщо ліміт вичерпано — спочатку видаліть зайве фото, потім додавайте нове
    - Товари без фото продаються значно гірше — не лінуйтеся додавати фото кожному товару

### 5.5. Варіанти, розміри і опції

Якщо у товару є розміри, кольори або інші опції — оформіть їх як **окремі варіанти**, а не вписуйте в опис вручну.

**Чому це важливо:**

- Коректний облік залишків по кожному варіанту
- Покупець бачить доступні опції і обирає потрібну
- Система не дозволить продати те, чого вже немає
- Статистика продажів по варіантах стає точнішою

### 5.6. Залишки і резерви

У товара є два важливі стани запасу:

| Поле | Значення |
|---|---|
| `stock` | Фактичний доступний залишок |
| `reserved` | Резерв під незавершені або ще не закриті замовлення |

!!! info "Що означає reserved"
    `Reserved` — це **не** продаж. Це тимчасове бронювання товару, поки замовлення ще не завершене. Коли замовлення оплачується, резерв стає реальним продажем. Якщо замовлення скасовується, резерв знімається і товар повертається у `stock`.

### 5.7. Режими складу

| Режим | Опис | Коли використовувати |
|---|---|---|
| `tracked` | Склад рахується, ви задаєте реальний залишок | Товари в наявності на складі |
| `untracked` | Жорсткий облік залишку не ведеться | Товари під замовлення |

!!! warning "Нагадування"
    Якщо товар іде `під замовлення` (untracked) — для нього потрібна хоча б одна **онлайн-оплата**. Тільки з післяплатою checkout буде заблокований.

### 5.8. Пошук, список і швидке редагування

У WebApp вкладка **Товари** призначена для щоденної роботи:

- Пошук за назвою або описом
- Перегляд усього списку каталогу
- Швидке редагування цін, активності, залишків
- Оперативні зміни без переходу в довгі сценарії

### 5.9. Імпорт каталогу

Якщо доступний модуль **Імпорт каталогу** — використовуйте вкладку `Імпорт` у WebApp.

**Що можна робити:**

- Отримати приклад CSV-шаблону в Telegram
- Отримати приклад XLSX-шаблону в Telegram
- Завантажити свій CSV або XLSX файл з товарами
- Переглянути звіт імпорту після завантаження

!!! example "Правила безпечного імпорту"
    1. **Спочатку скачайте шаблон** — не створюйте файл з нуля
    2. **Не змінюйте назви колонок** у першому рядку шаблону
    3. Замініть тестові приклади на свої реальні товари
    4. **Почніть з невеликого файлу** (5-10 товарів), а не з усього каталогу
    5. Після імпорту **перевірте кілька товарів вручну** — ціни, категорії, фото

### 5.10. Коли товар краще вимкнути, а не видаляти

Якщо товар тимчасово не продається — **зробіть його неактивним**, а не видаляйте. Це безпечніше, бо:

- Зберігається історія замовлень по цьому товару
- Товар можна повернути у каталог за один клік
- Не ламається аналітика

## 6. Замовлення, статуси та оплати

### 6.1. Які способи оплати підтримуються

Monobank Merchant — це єдиний повністю автоматичний спосіб оплати. Після оплати статус замовлення оновлюється **без вашої участі**. Для підключення вам потрібен **Merchant Token** від Monobank.

- Поточний статус
- Спосіб оплати
- Суму
- Дані клієнта
- Місто і точку доставки
- Склад замовлення (перелік товарів)

Для більшості власників саме Telegram-адмінка є основним місцем для оперативної роботи із замовленнями.

### 6.4. Основні фільтри замовлень

У Telegram-адмінці регулярно переглядайте:

| Фільтр | Що означає | Частота перевірки |
|---|---|---|
| `⏳ Очікують` | Нові замовлення, які чекають дій | Кілька разів на день |
| `🧾 На перевірці` | Клієнт натиснув «Я оплатив» — потрібна ваша перевірка | Кілька разів на день |
| `📦 Накладений` | Замовлення з післяплатою — треба готувати до відправки | Щодня |
| `✅ Оплачені` | Підтверджені оплати — готові до відправки | За потреби |
| `📋 Усі` | Повний список замовлень | Для пошуку конкретного замовлення |

!!! warning "Мінімальна звичка"
    Не накопичуйте замовлення в «Очікують» і «На перевірці». Перевіряйте ці фільтри **мінімум двічі на день**.

### 6.5. Сценарій онлайн-оплати

**Для Monobank Merchant:**

1. Клієнт оформлює замовлення
2. Система створює рахунок у Monobank і показує посилання на оплату
3. Клієнт оплачує
4. Monobank надсилає підтвердження автоматично
5. Замовлення переходить у статус **Оплачено** без вашої участі

**Для оплати за посиланням і реквізитами:**

1. Клієнт оформлює замовлення
2. Система показує посилання або реквізити
3. Клієнт переводить гроші і натискає **«Я оплатив(ла)»**
4. Замовлення переходить у статус **На перевірці**
5. Ви відкриваєте замовлення, перевіряєте оплату і **підтверджуєте**
6. Замовлення переходить у **Оплачено**

### 6.6. Сценарій післяплати

Для післяплати система використовує окрему логіку статусів:

| Статус | Значення |
|---|---|
| `📦 Накладений` | Замовлення створено, треба відправити |
| `🚚 Відправлено` | Посилка відправлена |
| `📍 Доставлено у відділення` | Посилка у відділенні Нової Пошти |
| `✅ Оплачено/Отримано` | Клієнт оплатив і забрав |

### 6.7. Що перевірити в оплаті до запуску

Для **кожного** ввімкненого способу оплати перевірте три речі:

1. Покупець **бачить** цей спосіб у checkout
2. Інструкція або платіжний крок працює **без** ручного пояснення менеджером
3. Ви можете **обробити** результат у замовленнях

!!! example "Чекліст по кожному способу оплати"
    - **Monobank Merchant**: Токен введений, оплата проходить, статус оновлюється автоматично
    - **Оплата за посиланням**: Посилання коректне, веде туди, куди треба
    - **Оплата за реквізитами**: Текст реквізитів повний і без помилок
    - **Післяплата**: Покупець бачить фінальну суму з урахуванням доставки

### 6.8. Якщо клієнт не оплатив або замовлення зависло

Не залишайте неоплачені замовлення без уваги. Це важливо, бо:

- Накопичуються резерви на товарах (інші не можуть купити)
- Аналітика і залишки починають відрізнятися від реальності
- Замовлення, які чекають тижнями, тільки засмічують список

Регулярно переглядайте фільтри **Очікують** і **На перевірці**.

### 6.9. Коли перевіряти кожне замовлення вручну

!!! tip "На старті магазину"
    Перші 2-3 тижні краще перевіряти **вручну кожне** нове замовлення, навіть якщо здається, що все налаштовано правильно. Це дозволяє:

    - Швидко зловити проблеми в текстах, цінах, оплаті
    - Перевірити, що клієнт отримує правильні повідомлення
    - Зрозуміти, де можуть виникати питання

### 7.3. Що можна робити з ТТН

Коли модуль налаштований, з картки замовлення можна:

- **Створити ТТН** — автоматично заповнить дані відправника з ваших налаштувань і дані одержувача з замовлення
- **Побачити номер ТТН** — для відстеження
- **Оновити статус** — перевірити поточний стан відправки
- **Скасувати ТТН** — якщо відправка не потрібна
- **Створити ТТН повторно** — якщо треба переробити

### 7.4. Що потрібно, щоб ТТН створилась без помилки

У самому замовленні мають бути заповнені **дані одержувача**:

- [ ] Місто одержувача
- [ ] Відділення або поштомат
- [ ] Ім'я одержувача
- [ ] Телефон одержувача

!!! warning "Часта помилка"
    Якщо ТТН не створюється — у 90% випадків причина в тому, що в замовленні не заповнені дані одержувача. Спочатку перевірте і виправте дані замовлення, а вже потім повторіть створення ТТН.

### 7.5. Коли використовувати «Скинути налаштування»

Кнопка скидання потрібна **не для щоденної роботи**, а тільки коли:

- Налаштування відправника введені з помилкою
- Ви повністю змінюєте реквізити відправки (інше місто, інше відділення)
- Ви міняєте API ключ

Якщо просто змінився телефон або ПІБ — оновіть ці поля і збережіть без скидання.

| Оплата за реквізитами | Увімкнути/вимкнути. Вписати текст реквізитів |
| Післяплата | Увімкнути/вимкнути |

!!! warning "Після будь-якої зміни в оплаті"
    **Відразу** робіть тестове замовлення. Зберегти — це не те саме, що перевірити. Клієнт не повинен бути першим, хто тестує вашу оплату.

### 8.2. Вигляд

Секція **Вигляд** керує темою WebApp-каталогу. Зміни впливають на каталог, checkout і загальний брендинг.

| Параметр | Що змінює |
|---|---|
| Тема | Світла або темна |
| Акцентний колір | Основний кольоровий акцент по всьому каталогу |
| Стиль карток | Як виглядають картки товарів |
| Шрифт | Гарнітура тексту |
| Скруглення | Наскільки «округлі» елементи інтерфейсу |
| Щільність | Як щільно розташовані елементи |
| Стиль кнопок | Форма і вигляд кнопок |
| Логотип | Картинка в шапці каталогу |
| Банер | Горизонтальне зображення у hero-блоці |

!!! tip "Поради щодо вигляду"
    - **Логотип** — квадратний формат, бажано PNG з прозорим фоном
    - **Банер** — горизонтальний, щоб акуратно виглядав у шапці каталогу
    - Після змін **завжди перевіряйте** вигляд каталогу: пошук, картки товарів, кнопки, hero-блок
    - Не змінюйте все одразу — міняйте по одному параметру і перевіряйте результат

### 8.3. Автоматизація

#### Нагадування про кошик

Якщо модуль доступний, ви можете налаштувати **автоматичні нагадування** покупцям, які поклали товар у кошик, але не завершили замовлення.

**Що можна налаштувати:**

- Увімкнути/вимкнути нагадування
- Змінити текст верхньої частини повідомлення
- Обрати інтервали нагадувань

| Інтервал | Коли надсилається |
|---|---|
| **30 хв** | Через 30 хвилин після покинутого кошика |
| **6 год** | Через 6 годин |
| **24 год** | Через 24 години |

!!! info "Як обрати інтервали"
    Для більшості магазинів достатньо **30 хв + 24 год**. Перше нагадування «ловить» тих, хто відволікся. Друге — тих, хто забув. Три нагадування підряд можуть дратувати покупця.

#### Модулі магазину

У цій частині ви вмикаєте функції, доступність яких залежить від тарифу:

- Нова пошта
- Промокоди
- Імпорт каталогу
- Нагадування про кошик
- Кампанії
- Клієнти та сегменти
- Аналітика
- Крос-продажі
- Кабінет покупця

!!! info "Якщо модуль не вмикається"
    Перевірте, чи ваш тариф підтримує цей модуль. Якщо тариф підтримує, але перемикач неактивний — зверніться до платформного адміністратора.

### 8.4. Операційка

#### Кабінет покупця

Якщо модуль доступний, можна:

- Увімкнути кабінет покупця
- Увімкнути відображення бонусів у кабінеті

**Що це дає покупцю:**

- Вхід без пароля (через Telegram)
- Прив'язку замовлень до e-mail
- Перегляд своїх замовлень і статусів
- Можливість бачити персональні бонуси або промокоди (якщо працює разом із модулем промокодів)

#### Нова пошта

Секція Нової пошти також знаходиться тут — керування API ключем і даними відправника. Детальна інструкція — в [розділі 7.2](#72-api).

### 8.5. Команда

У секції **Команда** ви керуєте списком адміністраторів магазину.

**Як додати нового адміна:**

1. Людина має виконати `/myid` у чаті з ботом
2. Вона має передати вам свій **числовий** Telegram ID
3. Ви додаєте цей ID у список адміністраторів

!!! warning "Обмеження по тарифах"
    - **Basic** — до 1 адміністратора
    - **Pro** — до 3 адміністраторів
    - **Business** — до 10 адміністраторів

    Додавайте тільки тих, кому **реально потрібен** доступ до замовлень, каталогу або налаштувань. Адмін бачить усі робочі дані магазину.

### 8.6. Тексти й кнопки

#### Тексти магазину

Тут редагуються всі тексти, які бачить покупець:

- Стартові тексти (привітання, опис магазину)
- Службові повідомлення
- Повідомлення в checkout та при оформленні

!!! tip "Після зміни тексту"
    Перевіряйте **живий сценарій** у боті та в каталозі. Не обмежуйтесь лише повідомленням «Збережено». Пройдіть шлях покупця і переконайтесь, що новий текст виглядає правильно в контексті.

#### Кнопки та назви

Тут змінюються підписи кнопок і назви елементів інтерфейсу.

Після перейменування кнопки — пройдіть повний шлях користувача ще раз, щоб переконатися, що новий текст **зрозумілий** і не ламає логіку сценарію.

## 9. Клієнти, кампанії, аналітика та модулі росту

!!! info "Коли вмикати ці модулі"
    Ці модулі потрібні **не для першого дня**. Вони стають корисними після того, як магазин стабільно приймає замовлення і є хоча б мінімальна клієнтська база.

### 9.1. Промокоди

У WebApp-вкладці **Промокоди** можна створювати знижки з такими полями:

| Поле | Що задає |
|---|---|
| Код | Текст промокоду, який вводить покупець |
| Тип знижки | Відсоток або фіксована сума |
| Значення | Розмір знижки (%) або сума (грн) |
| Мінімальна сума | Мінімальне замовлення для активації |
| Ліміт використань | Скільки разів можна використати (для всіх) |
| Опис | Пояснення для внутрішнього використання |
| Заголовок у кабінеті | Як промокод відображається покупцю |
| Сегмент | Для якої аудиторії діє |
| Місце показу | Де покупець бачить промокод |
| Активність | Увімкнений чи ні |

**Сегменти для промокодів:**

- Усі клієнти
- Нові (перше замовлення)
- Повторні
- VIP
- Sleeping (давно не купували)

**Де відображається промокод:**

- Тільки в checkout
- Тільки в кабінеті покупця
- І в checkout, і в кабінеті

!!! tip "Коли створювати промокоди"
    - Після того, як перевірені ціни, каталог і оплата
    - Коли ви розумієте, **кому саме** даєте пропозицію
    - Коли маєте чітку мету: перше замовлення, повторна покупка, повернення сплячих

### 9.2. Клієнти

У вкладці **Клієнти** ви бачите:

- Загальну кількість клієнтів
- Розподіл за сегментами (нові, повторні, VIP, sleeping)
- Середній чек
- LTV (lifetime value)
- Історію замовлень по кожному клієнту
- Останню активність

**Картка клієнта** допомагає зрозуміти: хто купує часто, хто давно не повертався, кому варто показувати окремі пропозиції.

### 9.3. Сегменти клієнтів

Сегментація потрібна, щоб **не надсилати всім однакові** кампанії і промокоди.

| Сегмент | Логіка |
|---|---|
| **new** | Клієнт ще не зробив жодного замовлення |
| **repeat** | Від 2 замовлень і більше |
| **VIP** | Від 3 оплачених замовлень або від 5000 грн виторгу |
| **sleeping** | Від 30 днів без активності |

!!! info "Пороги за замовчуванням"
    Базові пороги сегментів:

    - Повторний: від **2** замовлень
    - VIP: від **3** оплачених замовлень
    - VIP за виторгом: від **5000** грн
    - Sleeping: від **30** днів без активності

    На гнучкіших тарифах ці пороги можна налаштовувати.

### 9.4. Кампанії

У вкладці **Кампанії** ви можете:

1. Обрати **сегмент** для розсилки
2. Подивитися мета-інформацію по сегменту (скільки людей, середній чек)
3. Написати **текст повідомлення**
4. **Надіслати** кампанію
5. Подивитися результати відправок

!!! warning "Перед першою розсилкою"
    1. Перевірте сегмент — не надсилайте «всім», якщо не впевнені
    2. Перевірте текст на собі — надішліть тестову кампанію собі
    3. Не починайте з широкої аудиторії, якщо база вже велика
    4. Одна чітка пропозиція працює краще, ніж довгий текст з трьома акціями

### 9.5. Аналітика

| Період | Що бачите |
|---|---|
| Сьогодні | Оперативні дані за поточний день |
| 7 днів | Тижнева динаміка |
| 30 днів | Місячні підсумки |
| Весь час | Загальна картина від початку |

**Через аналітику ви відстежуєте:**

- Виторг
- Кількість замовлень
- Середній чек
- Конверсію
- Популярні товари
- Підсумкові інсайти

!!! tip "Ключові питання для власника"
    - Що продається краще за все?
    - Чи зростає середній чек?
    - Чи не падає конверсія?
    - Які товари дають основний виторг?
    - Чи є сезонні або тижневі патерни?

### 9.6. Крос-продажі

Якщо доступний модуль **Крос-продажі**, система автоматично показує пари товарів, які часто купують разом.

**Для кожної пари ви бачите:**

- Які два товари часто купують разом
- У скількох замовленнях вони були одночасно
- Частку приєднання до продажів кожного товару
- Сукупний виторг по парі

**Використовуйте для:**

- Добірок «З цим також купують»
- Ручних допродажів у чаті
- Промокампаній по парах товарів
- Покращення структури асортименту

## 10. Команда, доступи та підписка

### 10.1. Адміни магазину

| Тариф | Ліміт адміністраторів |
|---|---|
| **Basic** | До 1 |
| **Pro** | До 3 |
| **Business** | До 10 |

Перед додаванням нового адміна переконайтесь, що людина **реально потребує** доступ. Адмін бачить усі робочі дані магазину: замовлення, клієнтів, аналітику, налаштування.

### 10.2. Підписка

У блоці підписки можна:

- Побачити поточний тариф і його можливості
- Перевірити статус підписки та оплачений період
- Створити новий рахунок для продовження
- Відкрити останній рахунок
- Переглянути історію платежів
- Переглянути договори

!!! warning "Перевіряйте регулярно"
    Навіть якщо магазин давно працює — перевіряйте оплачений період підписки хоча б раз на тиждень. Якщо підписка закінчиться, магазин може перейти в режим обмеженого доступу.

### 10.3. Режими доступу до магазину

| Режим | Що відбувається |
|---|---|
| `auto` | Режим визначається логікою підписки автоматично |
| `full` | Повний доступ до магазину |
| `soft_freeze` | Каталог можна переглядати, але checkout заблокований |
| `frozen` | Магазин повністю недоступний для продажів |

!!! warning "Якщо магазин перестав приймати замовлення"
    Перше, що треба перевірити — це **режим доступу**. Якщо магазин у `soft_freeze` — клієнт не зможе оформити замовлення. Якщо у `frozen` — продажі повністю зупинені. Зазвичай це пов'язано із закінченням підписки.

## 12. Часті ситуації і що робити

### Магазин тимчасово недоступний

**Що перевірити:**

1. Статус підписки — можливо, закінчився оплачений період
2. Режим доступу — `soft_freeze` або `frozen` блокують продажі
3. Чи працює бот взагалі — якщо ні, це питання до платформного адміна

### Бот не відповідає

1. Перевірте, чи ви пишете правильному боту
2. Перевірте, чи є доступ до магазину
3. Зачекайте 2-3 хвилини — можливо, технічні роботи
4. Якщо бот не відповідає більше 10 хвилин — зверніться до платформного адміністратора

### Немає доступу до адмінки

1. Перевірте, чи ви під правильним Telegram-акаунтом
2. Виконайте `/myid` і порівняйте з тим ID, який додано в адміни
3. Попросіть власника магазину перевірити список адмінів

### Не вдається додати адміністратора

| Причина | Що робити |
|---|---|
| Неправильний Telegram ID | Попросіть людину виконати `/myid` ще раз |
| Досягнуто ліміт тарифу | Перевірте кількість адмінів на вашому тарифі |
| Немає потреби в доступі | Переглянте, чи дійсно людині потрібен адмін-доступ |

### Товар є в каталозі, але його не купують

- Чи товар **активний**?
- Чи є **фото**?
- Чи коректна **ціна**?
- Чи він у **правильній категорії**?
- Чи не заблокований checkout через налаштування оплати для товару під замовлення?

### Не проходить оплата

1. Перевірте, чи ввімкнений потрібний спосіб оплати
2. Для Monobank — чи коректний токен
3. Для оплати за посиланням — чи правильне посилання
4. Для реквізитів — чи повний текст реквізитів
5. Зробіть тестове замовлення після кожної зміни

### Не створюється ТТН

- [ ] Увімкнений модуль Нової пошти
- [ ] Збережений API ключ
- [ ] Обране місто відправника
- [ ] Обрана точка відправлення
- [ ] Заповнені ПІБ і телефон відправника
- [ ] В замовленні заповнені дані одержувача (місто, відділення, ім'я, телефон)

### Не видно промокодів, клієнтів або кампаній

Дві найчастіші причини:

1. Модуль не увімкнено — перевірте Налаштування → Автоматизація → Модулі
2. Тариф не дає доступу — перевірте ваш поточний тариф

### Аналітика або cross-sell не дають корисних даних

Це **нормально** на ранньому етапі. Ці модулі стають корисними після того, як магазин вже має достатньо замовлень і накопичені дані. Зазвичай — після 50-100 замовлень.

## 13. Що робити далі

Після базового запуску рухайтесь у такому порядку:

1. **Стабілізуйте** каталог і оплату — переконайтесь, що все працює без збоїв
2. **Вирівняйте** доставку і статуси замовлень
3. **Налаштуйте** тексти, кнопки та вигляд під свій бренд
4. **Підключіть промокоди** — коли є перші продажі і розумієте свою аудиторію
5. **Перейдіть до клієнтів і сегментів** — побудуйте базу лояльності
6. **Почніть кампанії** — розсилки по сегментах для повторних продажів
7. **Регулярно дивіться аналітику** — відстежуйте тренди і реагуйте
8. **Підключайте крос-продажі** — після накопичення достатніх даних

!!! tip "Головне правило"
    Не намагайтеся ввімкнути все одразу. Спочатку зробіть стабільною базову операційку: каталог, оплата, доставка, замовлення. А вже потім розширюйте магазин модулями росту.
